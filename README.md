# FirstWork — Compliance Service

---

## Prerequisites

- Docker
- Python 3.13
- uv
- Make
- Git

---

## Setup

```bash
git clone <repo-url>
cd FirstWork-Assignment
make setup
```

`make setup` does:
1. `uv sync` — create `.venv` and install dependencies
2. `docker compose up -d db` — start MySQL 8.0
3. Wait for MySQL healthcheck
4. `alembic upgrade head` — run migrations
5. Load `docs/seed_dump.sql` — insert seed data

---

## How to use

### Run the API
```bash
make start
```
Opens at **http://localhost:8000**.
Swagger UI: **http://localhost:8000/docs**

### Run the tests
```bash
make test
```
Tests run in-process via FastAPI's `TestClient` — **no need to start the server**. They do need MySQL running and seeded, which `make setup` already handled.

### Other commands

| Command | What it does |
|---|---|
| `make setup` | Bootstrap everything (venv + db + migrations + seed) |
| `make start` | Run the API with auto-reload |
| `make test` | Run all tests |
| `make migrate-new m="msg"` | Generate a new autogenerated migration |
| `make migrate` | Apply pending migrations |

## API reference

Base URL: `http://localhost:8000` · Interactive docs: `/docs`

All endpoints return JSON. `placeholders` is **always a list of dicts** — send `[]` for static rules, one dict for a single eval, many dicts for bulk eval.

### 1. Create rule — `POST /rule/`

**Base rule:**
```json
{ "name": "Adult", "field_name": "User.age", "operator": ">=", "value": "18" }
```

**Composite rule** (children referenced by id, comma-separated):
```json
{ "name": "AdultActive", "field_name": "__composite__", "operator": "AND", "value": "1,3" }
```

**Placeholder rule** (token in `value`):
```json
{ "name": "TplCountry", "field_name": "User.nationality", "operator": "==", "value": "{{country}}" }
```

| Status | Meaning |
|---|---|
| `201` | Created — returns the rule with its new `id` |
| `409` | Duplicate rule name |
| `422` | Bad operator / field / composite value |

---

### 2. Update rule — `PATCH /rule/{rule_id}`

Partial update — send only the fields you want to change.
```json
{ "value": "21" }
```

| Status | Meaning |
|---|---|
| `200` | Updated — returns the new rule state |
| `404` | Rule not found |
| `422` | Validation error |

---

### 3. Evaluate compliance — `POST /contracts/{contract_id}/evaluate`

*I Can Send List of Failure Base rules too*

**Request:**
```json
{
  "rule_ids": [14],
  "placeholders": [
    { "country": "IN", "age": "18" },
    { "country": "US", "age": "21" }
  ]
}
```
- `placeholders: []` → one evaluation with no bindings (for static rules).
- One dict → one evaluation.
- Many dicts → one evaluation per dict (bulk mode — satisfies the PDF's "list of tuples" requirement).

**Response (always the same shape):**
```json
{
  "evaluations": [
    {
      "placeholder": { "country": "IN", "age": "18" },
      "compliant": true,
      "rules": [ { "rule_id": 14, "result": true } ]
    },
    {
      "placeholder": { "country": "US", "age": "21" },
      "compliant": false,
      "rules": [ { "rule_id": 14, "result": false } ]
    }
  ]
}
```

| Status | Meaning |
|---|---|
| `200` | Evaluated — see each `compliant` flag |
| `400` | Missing placeholder binding / rule resolution error |
| `404` | Contract not found |
| `422` | Missing / invalid `rule_ids` |

---

### 4. Eligible contracts — `POST /companies/{company_id}/eligible-contracts`

For one company, returns the ids of contracts that pass the rule, per placeholder set.

**Request:**
```json
{
  "rule_id": 14,
  "placeholders": [{ "country": "IN", "age": "18" }]
}
```

**Response:**
```json
{
  "evaluations": [
    {
      "placeholder": { "country": "IN", "age": "18" },
      "eligible_contract_ids": [1, 3]
    }
  ]
}
```

| Status | Meaning |
|---|---|
| `200` | Returns the evaluations list (one entry per placeholder set) |
| `400` | Missing placeholder binding |
| `404` | Company has no contracts |

## Key decisions, trade-offs, and assumptions

### 1. Database design

**Requirement:** support every kind of rule — Level 0, Level 1 (AND / OR), and Level 2 (AND of ORs etc.) — using the same `rules` table structure (`field_name`, `operator`, `value`).

**Assumptions:**

1. A composite rule is stored with `field_name="__composite__"`, `operator ∈ {AND, OR}`, and `value` = comma-separated list of child rule ids (e.g. `"1,3"` or `"r1,r2"`). The children must already exist when the composite is created.
2. The same `rules` table holds basic, composites, and placeholder templates — distinguished only by what's in `field_name`/`value`. Example rows:

   | id | field_name | operator | value | name                                                 |
   |----|---|---|---|------------------------------------------------------|
   | 1  | `User.age` | `>=` | `18` | Adult *(base rule)*                                  |
   | 5  | `User.nationality` | `==` | `{{country}}` | TemplateNationality *(placeholder rule)*             |
   | 10 | `__composite__` | `AND` | `1,3` | AdultActive *(Level 1)*                              |
   | 13 | `__composite__` | `AND` | `11,12` | FullCompliance *(Level 2 — composite of composites)* |
   | 14 | `__composite__` | `AND` | `5,6` | TemplateDrivingAge *(placeholder composite)*         |

**Trade-offs:**

- **No unique index possible today.** A unique constraint on `(field_name, operator, value)` would prevent duplicate logical rules, but for composites the `value` is a list — `"1,3"` and `"3,1"` mean the same thing but compare as different strings. Skipped under time constraint.
- **Future enhancement:** add a `rule_hash` column that stores a hash of `(field_name, operator, normalized_value)` where composite `value`s are sorted before hashing (so `"3,1"` → `"1,3"` → same hash). Put the unique index on `rule_hash`. Combined with a create-time validator that sorts composite child ids, this collapses logical duplicates into one row.
- **Composite Rule `value` - can check foreign key against `rules.id` — We have to add validator on rule create/update + no DELETE(This is tricky, we have to store lookup using table/index instead of scanning all rules) endpoint to break references.

---

### 2. Rule engine(Graph Depth first search, it can support any level of rules)

**Requirement:** evaluate any rule (Level0, Level 1, Level 2) against a single contract 

**Compliance(AND check across every list of rules)**
When the API receives `rule_ids: [10, 13, 14]`, each id is treated as the root of its own rule graph. The contract is `compliant: true` **only if every root evaluates to true**. Each root can itself be composite — so `rule_ids: [13]` where rule 13 is `(11 AND 12)` and rule 11 is `(1 AND 2 AND 3)` collapses to: contract passes only if rule 1 AND rule 2 AND rule 3 AND rule 12 all pass. Effectively a Level 3 check (AND of Level 2 trees), even though stored rules cap at Level 2.

**Walk — depth-first, short-circuit.**
Each root(composite rule): if composite → split `value` on commas → recurse → `any()` for OR, `all()` for AND (both short-circuit on the first decisive child). `Level 0 Rule` do the actual `(field_name operator value)` comparison via `compare()`.

**Caching — per contract, per placeholder set.**
Two caches live on `RuleContext` (one instance per (contract , placeholder) pass):

- `rule_cache[rule_id]` → the DB row, so we never re-query the same rule.
- `output_cache[rule_id]` → the boolean result, so a shared child evaluates **once** even when many parents reference it. Example: rule `1=Adult` is reused by rules 9, 10, 11, 13, 14 — it runs once per contract no matter how many of those roots are evaluated.


**Support for look of failed rule**
The engine caches every child rule's bool during evaluation, so internally we already know which children failed inside a composite. The API currently returns only the top-level `{rule_id, result}` per requested root — the per-child breakdown stays in `_output_cache`

**Design — RuleContext, Placeholder Service.**
Two responsibilities, two classes:

- **`RuleContext`** (state) — holds the contract, the model map (`User`/`Company`/`Contract` → row), the rule cache, and the output cache. State only.
- **`PlaceholderService`** (behaviour) — pure `{{token}}` → value resolver. Held by `RuleEngine`, not by the context, since the same resolver can serve multiple engine runs while a context is single-use.

Engine itself is stateless: feed it a `RuleContext` + `PlaceholderService`, call `evaluate(rule_id)`, get a bool. Easy to test, easy to extend.

---

### 3. Code design

**Requirement:** four APIs — create rule, update rule, evaluate compliance, get eligible contracts — built around the single-table rule design above.

**Assumptions:**
1. **Update/Update rule — one at a time
2. **Evaluate compliance — a contract is compliant only if ALL supplied `rule_ids` evaluate true.** Each rule id can itself be Level 0, 1, or 2 — the engine recurses, so even though the API surface only supports up to Level 2 trees, evaluating a list of Level 2 rules is effectively a Level 3 compliance check (`(rule_13_passes) AND (rule_14_passes) AND ...`).
3. **Eligible contracts — returns only contract ids that pass the rule.

**Trade-offs:**

- **No bulk create/update.** Data integrity issue, they'd require cross-rule validation that's out of scope here.
- **`placeholders` is always a list in the API** — empty list = static rule, one dict = single eval, many dicts = bulk eval = [` {"age": 18, "country": "IN"}, {"age": 21, "country": "US"} `]
- **Validation lives in Pydantic, returns 422 (not 400).** FastAPI convention for schema-level failures. Handlers stay thin.

**Decision/Design**

- **`Rule.value` is always `string`.** Based on `FieldName` - ( e.g `User.age` ) retrieve  field data type and then casting logic applied to value on evaluation time.
- **`joinedload` on `user` + `company`** when fetching contracts — avoids 2N+1 in `/eligible-contracts`. `Reason` - during evaluate of rule, we need contract user, company instance, on lazy load it will hit `SELECT * From User ....`
- **`Indexes` - `Contract` - Create 3 Indexes, it's a relationship table of `User` and `Company`
- **Rule Create/Update there are some validators to basic only
- **`Code Modularized` - api, schema, services, models, repositories

## Tools and libraries

| Tool | Why |
|---|---|
| **FastAPI** | Auto Swagger docs, Pydantic-native request validation, async-ready, fast |
| **SQLAlchemy 2.x** | Mature ORM, `joinedload` for N+1 prevention, raw-SQL escape hatch when needed |
| **Alembic** | Standard migration tool for SQLAlchemy; supports autogenerate |
| **Pydantic v2** | Type-safe schemas + automatic 422 on validation failures; ~10x faster than v1 |
| **MySQL 8.0** | Production-grade RDBMS; widely supported; good with composite indexes |
| **pymysql** | Pure-Python MySQL driver — no native build needed; portable across machines |
| **uv** | Reproducible installs via `uv.lock`; faster than pip; built-in venv management |
| **pytest** | Standard Python test runner; works cleanly with FastAPI's `TestClient` |
| **Docker Compose** | One-line MySQL provisioning across dev machines |
